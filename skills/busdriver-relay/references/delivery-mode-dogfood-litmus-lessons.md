# Delivery Mode dogfood litmus lessons
> **HISTORICAL / SUPERSEDED — NON-PRODUCTION.** Current policy authority: repository-root `docs/coding-workflow-authority-map.md`; any later current-tense or positive-authority wording is retained evidence only and MUST NOT be treated as executable/current policy.

Use this when Hermes is asked to complete relay work end-to-end (commit/PR/merge) and the first Delivery Mode attempt is blocked by missing or stale evidence.

## User-correction signal

If the user says they used `/goal` or otherwise asked Hermes to "complete everything", do not treat `stale_or_missing` litmus/pre-PR evidence as the final answer. That is a midpoint. Hermes must dogfood the relay pipeline to produce the required evidence, or identify the precise missing Busdriver surface that prevents doing so.

## Operator workflow

1. Re-run Phase 0 and verify current repo/branch/dirty/lock/gate state from live tools, not summaries.
2. If the WIP is too large for one litmus pass, split it into logical staged groups and run commit-mode litmus per group.
3. Treat litmus FAIL findings as authoritative review feedback. Fix the blocker, stage the fix, run targeted tests, then rerun the same Busdriver litmus loop. Do not hand-wave a fail-closed state as completion.
4. Only after commit-mode litmus PASS should `hermes-busdriver-deliver --operation commit` run.
5. After all groups are committed, continue with post-commit verification, pre-PR dual review, push/PR, latest-head PR-grind, merge, and cleanup.

## Delivery executor hardening patterns discovered by dogfood

- Re-check delivery-status inside the finalization lock and ensure the repo root still matches the originally requested repo. A helper response must not be able to redirect mutation to another checkout before or after lock acquisition.
- Scrub Git-affecting environment variables for **every** authority check and mutating Git/GitHub subprocess, not only the final `git commit` / `git push` / `gh` call. In particular, `run_delivery_status` after lock, `git_output`, `staged_changes_present`, `diff_hash`, `git ls-remote`, and Busdriver trusted writer subprocesses must not inherit ambient `GIT_DIR`, `GIT_WORK_TREE`, `GIT_INDEX_FILE`, `GIT_CONFIG_COUNT`, or trace/config variables that can redirect repo/index/config semantics. For `bash`-invoked trusted writers, also scrub shell startup hooks such as `BASH_ENV`, `ENV`, and `ZDOTDIR`; otherwise a sanitized Git environment can still execute attacker-controlled shell code before the Busdriver script runs.
- A finalization lock with `acquired=true` but no non-empty token is not sufficient. Fail closed before any mutation because the lock cannot be reliably released.
- Backstop verdict diff hashes must bind to the exact candidate being reviewed. For pre-PR evidence, bind to the full **clean** branch diff against the PR base, not arbitrary staged/unstaged/last-commit/upstream-only diffs. Use Busdriver-compatible plain diff semantics: disable ext-diff/textconv/color, avoid repo/global attributes execution surfaces, and hash after stripping trailing newlines.
- `commit` evidence and `pre-PR` evidence are not interchangeable. `commit` with staged changes requires fresh commit-mode litmus; `pr_review_fresh` is tied to a clean committed branch candidate and must not authorize newly staged content.
- `pre-pr-review` must run only on a clean worktree. If the repo is dirty, fail closed before acquiring/using trusted writer semantics instead of running review scripts that cannot produce a valid branch-diff-bound marker.
- `pr-create` must require a clean worktree and verify that the PR head is exactly the reviewed local HEAD as it exists on the live remote. Comparing only local `@{u}` can be stale unless the implementation fetches or otherwise queries the remote branch head.
- `pr-create` must bind the fresh PR-review evidence to the same base/head it will ask GitHub to use: repo root, branch name, local HEAD, branch diff hash, PR base ref, and the live remote base commit. A marker for the same base name but stale local `origin/main`, another repo, another head, or another diff must fail closed.
- `--head` for PR creation must not name an unrelated branch. Require it to match the current branch unless a future explicit, tested cross-branch policy exists.
- Trusted writer subprocesses must fail closed on `TimeoutExpired` / `OSError` and return structured JSON artifacts instead of letting exceptions escape. Build their environment from a sanitized allowlist and an explicit state dir; do not inherit ambient `BUSDRIVER_STATE_DIR`. Treat caller-supplied plugin roots and env-derived plugin roots as untrusted for mutating writer execution; validate the resolved Busdriver plugin root against a fixed trusted install location or another explicit, repo-owned trust policy before invoking `init-review-loop.sh` / `run-review-loop.sh`.
- Normalize plugin paths used to invoke Busdriver trusted writers to absolute paths; existence checks in one cwd and execution in another must refer to the same files.
- Mutating operations that fail before the side effect, including delivery-status/authority failures, should still write a durable blocked run artifact carrying the run id and fail-closed reason. Do not reserve run artifacts only for successful or post-lock paths.

## Testing pattern

For each hardening rule, add a negative contract test proving the side effect did **not** happen (`git_commit`, `git_push`, `gh_pr_create`, marker write, or merge absent from side effects) plus a positive/current-hash test where appropriate. Include poisoned-environment tests (`GIT_INDEX_FILE`, `GIT_DIR`, `GIT_CONFIG_COUNT`, stale local remote refs), stale-marker/hash tests, and clean-worktree precondition tests. Keep the main Hermes role as operator/verifier: subagents or Codex review can find blockers, but Hermes must re-read and verify before committing/pushing/merging.

## Large litmus-loop operations

- If a staged candidate reaches Busdriver's size or runtime limits, split it by commit candidate rather than increasing limits indefinitely. In practice, separate implementation (`scripts/...`) from regression tests (`tests/...`) when combined diffs make prompt construction or Codex review stall. The split is valid only if each candidate remains coherent and the follow-up test commit gets its own commit-mode litmus PASS.
- Do **not** split files that form one evidence/gate contract just because the directory-based suggestion says so. If `deliver` enforces a marker field produced by `litmus-status`, stage the producer (`scripts/hermes-busdriver-litmus-status`) and consumer (`scripts/hermes-busdriver-deliver`) plus their paired contract tests together, or commit the producer first. A litmus finding like `commit_litmus_staged_diff_hash_missing` after `commit_litmus_fresh` usually means the group omitted the status/marker summarizer that exposes the hash.
- If implementation-only staging makes the current targeted tests fail because the test file already encodes the new behavior, that is a regrouping signal, not a reason to ignore tests. Restore/stage the paired tests or make a smaller coherent producer-first group, then rerun targeted tests before litmus.
- If `run-review-loop.sh` reaches max iterations or a foreground Hermes call times out, do not raw-write PASS markers. Use Busdriver's own `init-review-loop.sh --force <n>` to restart the loop when appropriate, or reduce/regroup the staged candidate. A stale or missing `.claude/litmus-passed.local` remains a hard stop for commit.
- Background process execution is useful for long litmus reviews, but if the process sits with only the shell parent alive and no Codex child/progress for several minutes, inspect process state and prefer a smaller staged group over waiting blindly.
- After splitting staged groups, re-check `git status --short` and staged vs unstaged diffs before every litmus run. `git restore --staged <file>` used to isolate one group can leave necessary regression tests unstaged for the next group; do not assume both implementation and tests are staged.

## Additional hardening cases from delivery dogfood

- Redact finalization lock tokens from every emitted artifact/envelope, including `lock_acquire`, `lock_release`, nested command argv, and any key whose name contains `token`, `secret`, `password`, `passwd`, or `api_key`. A lock token is a capability to release another operation's single-flight guard and must not appear in durable JSON.
- Use the same state directory for finalization locks and Busdriver marker/evidence checks. When `--busdriver-state-dir-name` is provided (for example `.opencode`), acquire and release the finalization lock under that state dir instead of silently falling back to a default `--state-dir`/`.claude` location.
- Compute diff hashes over raw bytes, not `text=True` decoded strings. Keep Busdriver-compatible trailing-newline stripping (`rstrip(b"\n")`) but avoid Unicode decoding so binary/non-UTF-8 diffs neither crash nor hash different bytes than Busdriver evidence.
- `push` should set upstream (`git push -u origin HEAD:<branch>`) so the normal push → PR-create flow has a branch/upstream relationship for later verification. Still verify live remote head at PR creation time; upstream setup alone is not proof the remote is current.
- `merge` must be bound to the PR head that was just checked by PR-grind. Require a valid `latest_head` from the clean PR-grind payload and pass `gh pr merge --match-head-commit <latest_head>`; if the clean payload lacks a head SHA, fail closed instead of merging a moving target.
- `push` must bind `pr_review_fresh` evidence to the exact candidate it will publish, just like `pr-create`: repo root, current branch, current HEAD, PR base ref, branch diff hash, and live remote base commit. A stale/unrelated PR-review marker must not authorize `git push` merely because its decision status says `pr_review_fresh`.
- Validate the required read-only backstop verdict **before** running Busdriver pre-PR review scripts. Missing, unreadable, malformed, or candidate-mismatched verdict files should fail before `init-review-loop.sh` / `run-review-loop.sh` can mutate marker/review state.
- Backstop verdict files should be structured artifacts, not arbitrary `{status: PASS, diff_hash: ...}` JSON. Require schema/version, `read_only=true`, an independent backstop role, a nontrivial summary, reviewed repo root, reviewed HEAD, reviewed base ref, and current candidate diff hash.
- Do not expose no-op mutating operations in `argparse` choices. If a surface such as autonomous `pr-grind-fix-loop` is policy-blocked or not implemented, keep it out of the executable CLI (or behind a clearly read-only/status surface) until it has real gated semantics and tests.
- Handle finalization lock release failures without losing side-effect truth. If `git commit` / `git push` / `gh pr create` / `gh pr merge` already succeeded but lock release fails, return a fail-closed status such as `committed_release_failed` / `pushed_release_failed` while preserving the completed side effect in the artifact; do not report plain success, and do not hide the side effect as if nothing happened.
- Avoid mutable placeholder evidence in blocked artifacts. Only attach a release-placeholder object to envelopes for successful side-effect statuses that genuinely need post-`finally` release evidence; pre-side-effect blocked returns should carry `lock_release: null` or explicit release evidence, not an object later mutated by `finally`.
- Align state-dir precedence across delivery-status, finalization locks, and Busdriver trusted writers. Explicit `--state-dir` should win over env-derived/default Busdriver state dir; otherwise the lock may be checked in one state dir while markers are read/written in another.
- If `deliver commit` requires staged-diff evidence, `litmus-status` must not report `commit_litmus_fresh` for marker formats that cannot surface a `diff_hash`. Either parse and expose the hash from the marker or fail closed earlier as stale/missing; do not let the executor pass freshness and then block every commit with `commit_litmus_staged_diff_hash_missing`.
- Match Busdriver staged-diff hash semantics exactly when comparing commit evidence: hash raw `git diff --cached` bytes after Busdriver-compatible trailing-newline stripping (`rstrip(b"\n")`). Avoid decoded text hashes, but do not include final newline bytes if the trusted writer strips them.
- Do not paper over default porcelain untracked directory entries like `?? .claude/` by allowlisting the whole state directory. Prefer making status collection use `git status --porcelain=v1 -uall` (or equivalent) so untracked marker files are enumerated individually; otherwise a directory-level allowlist can hide unrelated unreviewed payloads.
- Marker dirty allowlists should distinguish evidence-only worktree marker writes from staged marker changes. Worktree-only trusted-writer marker updates may be ignorable for preflight cleanliness, but staged marker additions/modifications are not bound into `base...HEAD` or staged candidate hashes and should block push/PR/commit gates.
- Treat Busdriver trusted writer plugin roots as content/version trust, not just path trust. A path under `~/.claude/plugins/marketplaces/busdriver` is still unsafe if litmus scripts or `package.json` are symlinked, untracked, or dirty; verify required files are present, tracked when in a git checkout, non-symlink, and clean before executing `init-review-loop.sh` or `run-review-loop.sh`.
- For gated `push`, re-check the live remote branch immediately before the mutating push and use an expected-OID lease such as `git push --force-with-lease=refs/heads/<branch>:<remote_head> origin HEAD:<branch>`. Treat lookup failures other than “remote branch not found” as blockers; the prior PR-review evidence check is not sufficient against a concurrent remote update. If preflight reports `remote_branch_not_found`, handle the first-push race explicitly: a failed push should re-query the remote and report a specific `remote_branch_changed_before_push`/similar concurrent-change reason if the branch appeared, rather than collapsing to generic `git_push_failed`.
- Initialize variables used by `finally` cleanup (especially `lock_state_dir`) before the `try` block and track whether they were initialized. A cleanup path that references a variable assigned only inside the `try` can raise `UnboundLocalError`, mask the original failure, and skip lock release.
- Treat PR bases as origin-bound unless the delivery policy explicitly supports another remote. If a user-supplied base looks like `<remote>/<branch>` and `<remote>` is not `origin` but exists as a configured remote, fail closed instead of passing `upstream/main` to `git ls-remote origin refs/heads/upstream/main`.
- When validating untrusted JSON paths such as `reviewed_repo_root`, wrap `Path(...).expanduser().resolve()` in `try/except (OSError, ValueError)`. Embedded NULs or invalid path values should become structured mismatch blockers (for example `backstop_verdict_repo_mismatch`), never uncaught delivery-command crashes.
