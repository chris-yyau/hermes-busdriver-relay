# Exact-tree delivery and CI truth review lessons
> **HISTORICAL / SUPERSEDED — NON-PRODUCTION.** Current policy authority: repository-root `docs/coding-workflow-authority-map.md`; any later current-tense or positive-authority wording is retained evidence only and MUST NOT be treated as executable/current policy.

Use this reference for read-only final-delivery reviews where the authority is an exact Git tree rather than the current branch or worktree.

## Exact identity without repository mutation

- Inspect the named tree object directly; do not assume `HEAD`, the index, or a linked worktree is authoritative.
- If live external-index reconstruction from the mutable worktree yields a different tree than the named delivery tree, treat that as ambient drift/scope evidence rather than a disproof of the named artifact. Stop using live worktree lines as final evidence; materialize the named tree into scratch and bind all review/test citations to that materialization.
- Independently recompute the root tree from a previously extracted snapshot by hashing Git blob and tree payloads in memory (`sha1("blob <len>\0<bytes>")`; tree entries are raw-name byte sorted). This verifies identity without `git add`, a temporary index, or writing Git objects.
- Corroborate the result against any delivery commit's `^{tree}`, but do not treat commit reachability as the independent recomputation. If the commit object is unavailable locally but live CI is being cited, use the hosting provider's read-only Git-commit endpoint to obtain the commit's tree ID and require it to equal the independently recomputed reviewed tree before attributing check runs to that tree. An exact commit supplied by the caller makes a PR lookup unnecessary; query the immutable commit/check surfaces directly.
- Seal the reviewer footprint by comparing ending HEAD, index digest, porcelain-status digest, refs/config, cache count, and—when strict object-store immutability is part of the contract—row-level object-directory metadata plus a separate semantic/content manifest. Retain the rows, not only one aggregate digest, so any drift can be localized. If the session is interrupted before the second fresh reseal and footprint comparison, report `no PASS` and separate completed probes from omitted verification.
- Reserve an explicit endgame tool budget before optional cross-checks: pristine-tree recomputation, closing source/candidate seal, footprint comparison, and final live-CI refresh are mandatory tail operations. When an iteration cap is near, run those before exploratory manifest scripts or extra diagnostics, then perform no more candidate operations. A successful test matrix cannot compensate for a missing close seal.
- Build the deterministic close harness at opening time, not at the end: it should recompute the pristine tree, compare opening/closing row manifests, refresh immutable-SHA CI evidence, scan for caches, remove every authorized scratch path, and verify absence in one bounded call. Dry-run its read-only comparison logic before broad tests. Never spend the last available interaction writing an audit script that has not yet executed; an unexecuted checker is not evidence.
- Treat approval-pending commands as outstanding side effects. Do not continue preparing or testing the same scratch path while a delayed command could still replace or delete it. Resolve or cancel the pending action first, or abandon those pathnames and use a fresh unique scratch generation. Re-verify scratch Git metadata immediately before each Git-dependent test batch so a late scratch rewrite is classified as harness interference rather than a candidate failure.
- Establish a hard endgame checkpoint before any optional long-running suite or new independent checker. At that checkpoint, require enough remaining interactions for: collect result, close reseal, footprint/evidence comparison, scratch deletion, and final response. If that reserve is uncertain, close immediately and omit the optional probe; report omitted breadth rather than sacrificing attestation.

### Keep external-index Git state hermetic

- Scope `GIT_INDEX_FILE`, `GIT_OBJECT_DIRECTORY`, `GIT_ALTERNATE_OBJECT_DIRECTORIES`, `GIT_DIR`, `GIT_WORK_TREE`, and process-local `GIT_CONFIG_COUNT`/`GIT_CONFIG_KEY_*`/`GIT_CONFIG_VALUE_*` overrides to a subshell or to each individual Git command. Never leave them exported for later test processes; Hermes terminal environment can persist between calls.
- Before replaying tests that initialize temporary repositories, explicitly unset all of those variables and run a tiny `git init`, `git commit`, and `git rev-parse --git-dir` probe in scratch. Mass commit/object/repository failures across unrelated tests are evidence of harness contamination until a clean-env rerun proves otherwise, not candidate defects.
- Serialize source observation and executable tests. Contract suites may deliberately move, sync, or replace repository files and Git metadata before restoring them; a tree reconstruction or reviewer read performed concurrently can see a mixed transient generation. Run tests in an isolated exact-tree materialization, wait for them to exit, then perform a fresh external-index reseal. Never use concurrent source reads as exact-tree evidence.
- When the source worktree is sparse or only partly materialized, its external-index `write-tree` can still be exact while named paths are absent on disk. Materialize the named tree from the object database into an isolated Git worktree before running workflow commands. If tests depend on the candidate being dirty relative to its authority base, keep scratch `HEAD` and the scratch real index at the base and overlay the complete candidate filesystem with a separate temporary index/archive; verify zero staged paths plus `write-tree == target` before and after.
- If tests need Git metadata beside a materialized target, initialize only a scratch-local `.git` after clearing ambient Git state, then verify its `write-tree` equals the reviewed tree. A scratch init performed while `GIT_OBJECT_DIRECTORY` or `GIT_INDEX_FILE` is inherited can appear to work while placing state in the reviewer's external store.
- Keep every index, object directory, materialization, venv, and cache below the designated runtime scratch root. Under a strict no-object-store-metadata-mutation contract, capture the original object-store footprint before target resolution, then make private byte copies (never hardlinks) of source, candidate, and replay object databases. Route every object-aware Git command through those copies; do not expose the originals as alternates merely because the command is logically read-only.

### Distributed object graphs and disposable clean-tree checks

- A linked worktree's `.git` file is only a pointer. Resolve its worktree-admin directory and `commondir` from file bytes, then seal the linked-worktree admin area, common Git directory, object store, refs, and config **before** any object-consuming command such as `rev-parse HEAD^{tree}`. If this closure is discovered late, establish a fresh opening seal before substantive review and disclose that the first footprint was incomplete.
- A deliberately dirty source worktree can still be independently tree-addressed without `git add`: parse a byte copy of its index for stage-0 path/mode rows, overlay current tracked path bytes and modes, add the explicitly observed non-ignored candidate paths, reject unexplained extras/caches, and rebuild blob/tree hashes in memory. Require this source reconstruction and the pristine exact-tree materialization to produce the same root tree and semantic-map digest.
- Do not assume the target tree, top commit, parents, and unchanged blobs live in one replay store. Exact-tree deliveries may place the root tree in a post-test reseal store, commits in an independent replay store, and inherited objects in the common store. Locate each required object, make byte copies of every contributing store under scratch, compare source/copy semantic manifests, and expose only those private copies through a scratch-local bare `GIT_DIR`, `GIT_OBJECT_DIRECTORY`, and `GIT_ALTERNATE_OBJECT_DIRECTORIES`.
- `git cat-file` still needs a valid `GIT_DIR` even when object-directory variables are set; initialize an empty private bare repository rather than pointing at protected Git metadata.
- With a private bare repository, `checkout-index` needs an explicit scratch `GIT_WORK_TREE`. After `read-tree`/checkout, an unrefreshed disposable index can make `diff-index --quiet` report every path as modified solely because its stat cache is empty. First independently rehash the materialized paths/modes/bytes to the expected tree, then run `git update-index --refresh` on the **disposable private index** and require both worktree and cached diffs to be clean.
- Treat optional external-release downloads and semantic probes as pre-endgame work. Once mandatory evidence rows are green, close immediately; do not spend the reserved seal/cleanup interactions on a large optional download.

### Close correctly when the live worktree moves

- If the source worktree changes after the opening seal, reconstruct its new tree with a second fresh external index and record the changed-path set. Treat that result only as concurrent-drift evidence; do not mix its lines or test outcomes into the named-tree review.
- Close-seal the preserved named-tree materialization separately with another fresh external index/object directory and an independent in-memory tree hash. This can preserve attribution to the named artifact, but it does **not** justify claiming that the live source footprint stayed unchanged.
- Report both truths explicitly: the reviewed snapshot remained the named tree, while the live worktree moved. Also state whether source HEAD, source index, refs, configs, and cache counts stayed fixed; never silently relabel concurrent drift as reviewer mutation or as target-tree content.

## Mutation-outcome truth is compositional

A correct transport reconciliation branch is not enough. Test every later failure boundary with the uncertain outcome injected.

- For a failed publish command followed by an unrelated live HEAD, never claim that live HEAD as this invocation's commit. Preserve an explicit `outcome_unknown` state unless authoritative postconditions prove the candidate landed or did not land.
- Force the durable-artifact writer to fail after an `outcome_unknown` executor result. The stdout fallback must retain the unknown status, the nested mutating transcript, and reconciliation requirements. It must not collapse to an ordinary pre-side-effect `artifact_write_failed` result or clear `mutating_run`.
- Keep known-completed, known-no-effect, and unknown-outcome artifact-write fallbacks separate. Persistence failure cannot erase a side effect or uncertainty about one.

## Required-check truth and provenance

- Compare the policy lock with the live branch-protection API as sets of `(context, app_id)`, including legacy contexts. GitHub may return the same app-bound requirement in both modern `checks` and legacy `contexts`; normalize modern rows first, then add `(context, null)` only for legacy names absent from the modern set. Naively unioning both surfaces creates false duplicate drift, and directly sorting mixed `null`/integer app IDs can crash the close harness. A passing job with a renamed context still does not satisfy the old required context. Preserve and print the normalized rows used for the decision.
- Prove that reporter-identity regression tests are themselves enforced by a required PR lane. The mere presence of a test such as “every lock row has the expected non-null `app_id`” is not CI protection when the required workflow's portable allowlist excludes that test file. On a scratch copy of the exact tree, replace the locked app IDs with `null` (or another invalid value) and run the **exact command block used by the required job**; the lane must fail. If it still passes, report that required CI can remain green while reporter provenance drifts. Extract or copy the workflow command verbatim and preflight every selected path—do not hand-transcribe a long allowlist, because testing a different command proves nothing about the required lane.
- Normalize the **exact-tree lock bytes first**, before trusting a handoff claim that the lock is “app-bound.” Print the complete pair sets. A GitHub Actions check run reported by app `15368` does not repair an exact-tree lock row whose missing `app_id` normalizes to `null`; that candidate is rollout-incompatible and the production checker should report drift.
- Treat `source_app`, workflow path, and job ID in a lock as descriptive unless the verifier actually enforces them. If GitHub branch protection is app-bound, preserve the app ID in the lock; do not make the server match a null-app lock by weakening protection.
- Probe workflow/job mapping with a nonexistent regex-shaped **job ID** such as `.*`, not only a metacharacter-bearing display/context name. A shell line parser that interpolates the job ID into AWK/grep can select an unrelated real job and false-clean when its display name matches. Require literal key lookup or a real YAML parser, and retain the executable negative test.
- When delivery authority requires actual scanner execution, accept only `COMPLETED/SUCCESS` for every required context. GitHub may treat `SKIPPED` or `NEUTRAL` jobs as satisfying branch protection, but that proves routing/policy acceptance—not that the scanner ran. Required scanner jobs must not be behind path classifiers or job-level skip conditions; either run them on every PR or require a separately trusted aggregate context whose contract explicitly rejects skipped children. For a per-scanner-success requirement, only the former is sufficient. Add a portable source-contract regression that forbids `needs`/`if` skip gates on the required job IDs, and verify doc-only stack tips as well as code tips.
- Distinguish repository source correctness from rollout state. A locally consistent lock can still leave the PR blocked until live branch protection converges.
- Re-read live CI at final reporting time. Runs can be rerun or replaced while review is in progress; report attempt/job IDs and whether each lane is queued, executing, completed, or failed.
- An atomic push of a stacked PR chain can update a PR head and then its base in quick succession. GitHub may create two `pull_request` runs for the same PR and exact head SHA, cancel the first, and queue a replacement. `statusCheckRollup` can retain both generations in arbitrary order, so neither “any cancelled check means failure” nor a last-write-wins dictionary is admissible. Group by `(PR, head SHA, workflow name)`, select the newest run by `(created_at, run ID)` (or newest `startedAt` when only rollup rows are available), and bind required check runs to those selected run IDs via `details_url`. A stale cancellation is superseded only when a newer exact-head replacement exists; wait for that replacement and require its required jobs to be `COMPLETED/SUCCESS` from the expected App. If the newest generation is terminal non-success or no replacement exists, fail closed. Preserve stale and selected run IDs plus the selection reason in evidence.
- If a temporary CI PR moves away from the reviewed SHA during the review, separate immutable exact-SHA check-run evidence from current-PR evidence. The old check runs may still prove execution for that SHA, but the PR must no longer be described as currently headed by the reviewed commit.

## Scratch-only API evidence

- A mandated scratch root covers reviewer tooling side effects too, not only test outputs. Before using extraction helpers, account for implicit cache files. Prefer stdout-only `gh api`/`curl` queries (or explicitly route the helper cache under the approved root) when a web extractor would spill large-response caches under the normal home directory. `web_extract` may persist a truncated response under `~/.hermes/cache`; if it is used anyway, record the exact emitted cache path, remove it during the same review, and verify absence at close.
- Record any unavoidable tool-managed cache outside the approved root as a footprint exception; never silently claim “scratch only.”

## Ubuntu and self-hosted lanes

- Prove both configured routing and observed execution when making a live runner-trust claim. Inspect the exact-tree workflow's `runs-on` value, then query the exact-SHA Actions run/jobs endpoints and retain `head_sha`, event, attempt, job IDs, labels, `runner_group_name`, status, and conclusion. A workflow label alone proves only requested routing; an observed `runner_group_name: GitHub Actions` plus `ubuntu-latest` labels proves that the cited run actually used the hosted pool. Bind that run to the reviewed tree through the commit's tree ID first.
- A small GitHub-hosted Ubuntu allowlist establishes portability only for that allowlist. Name the boundary; do not generalize it to the full contract suite.
- A self-hosted runner label or an `ephemeral` runner name is routing metadata, not host attestation. Do not claim cross-host or host-identity proof without an independently selected and verified attestation model.
- A `pull_request` workflow that checks out candidate bytes and runs candidate-controlled tests on a trusted self-hosted host is a code-execution trust boundary. Require verifiable ephemeral isolation/no-secret provisioning, or arrange for a trusted harness to treat candidate bytes as data.
- Keep a queued/missing-runner condition distinct from an executed test failure and from successful host evidence.

## Independent manifest replay

- Treat manifest key namespaces as schema, not as universally repository-root-relative paths. Before hashing rows independently, derive and record the base for each section (for example repository root, `scripts/`, a plugin/source root, or an external unavailable root) from the production consumer and contract tests. A missing file under an assumed base is first a checker-model error, not candidate drift. Fail only after the section's authoritative path semantics are established.
- Keep the independent checker independent of the manifest's claimed inventory, but not ignorant of its schema: derive executable surfaces from exact-tree modes, recompute bytes from the correct section roots, and compare both directions. Run this optional replay before the reserved closing-seal tail so a checker bug cannot consume the calls needed for attestation.

## Exact suite execution and exit authority

- Derive the required test command from the **exact-tree workflow bytes** on every generation. Parse the actual command tail and assert every selected path exists in the named tree; never reuse a hand-copied allowlist from an older candidate even when its test count looks plausible.
- On macOS, wrap piped pytest logging in an explicit `/bin/bash` script with `set -o pipefail`, capture `rc=${PIPESTATUS[0]}`, append that value to the log, and exit with it. Do not rely on the terminal's default shell: zsh and Bash expose pipeline status differently, so a wrapper can report exit 0 after real pytest failures.
- A wrapper process exit is authoritative only after the log's pytest summary, captured pytest exit, and process exit agree. Hash the final log and bind it to a post-test exact-tree reseal.

## Smoke and cache adjudication

- Verify both mechanism and postcondition: use `-B`/cache-provider disabling/in-memory `compile()`, then scan the source snapshot for `__pycache__`, `.pytest_cache`, `.pyc`, and `.pyo` after replay.
- Cache hygiene and temporal viability are separate. Compare the smoke deadline with the measured exact-tree suite duration. A clean source tree is not a passing smoke contract if the default timeout is shorter than the observed suite runtime.

## Durable artifact classification

- Separate durable bytes from durable authenticated evidence. An artifact protected by a process-local HMAC key can survive on disk but cannot authenticate in a fresh process or another host.
- If documentation explicitly says fresh-process lookup returns `run_not_found`, classify the artifact as same-process authenticated storage, not a resumable cross-process state machine or signer/broker capability.
- Do not infer host attestation, cross-process authority, or reusable finalization authority from filesystem durability alone.
